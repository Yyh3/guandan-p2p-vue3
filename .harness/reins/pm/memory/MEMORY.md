### PM写修复 spec 前必须审计4 个维度的副作用(2026-06-09)
Type: workflow

**触发场景**:写「修某模块 bug」的 spec,owner 给了一组 bug +修复方向,但没审计「修复方案在调用方 /注入点 /资源清理 /测试可行性4 个维度的副作用」。

**4 个维度**(在动笔写 spec 前必查):

1. **调用方时序依赖** ——读「被改模块」+「所有调用方」+「模块自己的状态初始化时序」,检查修复在不修调用方前提下能否独立成立
2. **数据注入点** ——修复需要新增字段(如 UUID),检查走「调用方传入」还是「模块内部自动注入」 —— 后者更鲁棒(避免调用方漏传)
3. **资源清理** ——修复引入新的定时器 /监听 / handle,检查 close / dispose / onUnmounted 是否清理 —— 否则反复进/出内存泄漏
4. **测试可行性** ——修复依赖时间 /异步 /副作用,检查测试框架能否在合理时间内验证 —— 否则要设计 fake timer / mock方案

**必做动作**(每个维度都过一遍,任何「不修就修不通」的发现都写进 spec):
1. §〇「PM关键提示」——顶部加警示框(读 spec 前先看)
2. §二「改动文件清单」—— 加进文件列表
3. §X(单独一节)——给出最小 diff
4.同一信息在3 个地方重复,防止 coder跳读漏看

**「别动 X以外的文件」这种约束,如果跟硬依赖冲突,要在 spec 里显式打破并写明论证** —— 不打破就修不通,硬约束就是骗自己的。

**反例 A**(P0,2026-06-09 上轮):只看 network.js内的3 个 P0修复,没看 RoomView.vue,以为改完3 个函数就够了。结果4-tab永远不通,因为 host 的 BroadcastChannel名字固定是 default、joiner找的是123456(setRoomId顺序错)。

**反例 B**(P1,2026-06-09 本轮差点犯):只看 network.js内的3 个 P1修复方向,以为直接照搬就行。结果发现:
- UUID 如果走调用方传参 → RoomView.vue L185 不传 uuid →字段缺失 → 必须改 RoomView(违反「别动 View」约束)
- 心跳 setInterval 不在 close() 里清 →反复进/出房间内存泄漏
- 测试用真实时间 → 心跳超时10s +撞座 retry300ms跑十几个测试要几分钟 → 测试套件挂死

3 个副作用都在写 spec 时发现并写进 §〇顶部,最终 coder 只要按 spec走就一次到位。
