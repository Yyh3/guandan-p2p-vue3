##〇、PM关键提示(读 spec 前先看)

修这3 个 P1 时,PM审计发现 **2 个 owner 没列出的强相关副作用**——必须同步修,否则3 个 P1 bug任何一个都修不彻底:

> **副作用 A — UUID注入必须发生在 network.js内部,不能依赖调用方传入**:
>
> 当前 `RoomView.vue` L185 调用 `net.joinRoom(roomNo, { nickname, avatar })` **不带 uuid字段**。
> 如果 UUID走调用方传参,需要改 RoomView.vue + GameView.vue + 所有未来调用方——破坏「别动网络层以外的文件」约束,且新增业务时容易漏。
> **正解**:`joinRoom` / `startAsHost`内部从 sessionStorage 自动注入 uuid,调用方零改动。

> **副作用 B — 心跳 interval必须在 close() 里清,否则内存泄漏**:
>
> joiner `setInterval(3000)` 发心跳,host `setInterval(5000)`扫超时。
> 当前 `close()` 只关 channel,**不清 interval**——单局跑完不致命,但反复进/出房间会泄漏。
> `RoomView.vue` L195 `onUnmounted(() => net.close())` 已经调 close,所以只要 close()内部清就行,**RoomView不用改**。

> **副作用 C — P0 的 MockBroadcastChannel 不支持 fake timer**:
>
> `network.test.js` 的 `MockBroadcastChannel.postMessage` 用 `Promise.resolve().then(...)`模拟异步派发,但**心跳 / 并发撞座 / 心跳超时都依赖时间**(3s /5s /10s /300ms),真实跑要等十几秒,测试套件挂死。
> 正解:`network.js`内部把 `setInterval` / `setTimeout`抽到 `_setIntervalFn` / `_setTimeoutFn` 模块级变量,默认指向原生 API;测试通过内部 reset 函数替换为可控的 fake timer,直接调 callback模拟「时间走过」。

> **副作用 D — GameView.vue 不调 net.close()**(已知,不在本轮):
>
> `GameView.vue` L846 `onUnmounted` **不调** `net.close()`(只清理 NICK_UPDATE / keydown / timer)。
> 影响:RoomView → GameView → 回首页时,RoomView 已 unmount调过 close,channel早关,所以正常路径没泄漏。
> 但如果用户先开房间,RoomView 不 unmount(被 Vue Router缓存)就跳 GameView,channel 关不了。
> **本轮不修**(属于 RoomView/GameView路由生命周期问题,留给后续 plan)。

**这4 个副作用里 A/B/C 是本轮硬依赖**——已写进「改动文件清单」、「测试要求」和「验收标准」。

## 三、3 个 P1 的具体改法

###3.1 Bug1 — UUID 去重(P1 高)

**根因**:
- joiner刷新 tab 后重新 join,host不知道是同一个人,分配新 seat →旧 seat永远占着
-一个人占2 个座位,挤掉真玩家

**修复前**(host收 JOIN 时,直接找空位分配):
```js
if (msg.type === 'JOIN') {
 const used = new Set(Array.from(peers.keys()))
 let assignedSeat = -1
 for (let i =1; i <4; i++) {
 if (!used.has(i)) { assignedSeat = i; break }
 }
 if (assignedSeat === -1) { sendMessage({ type: 'ROOM_FULL', ... }); return }
 peers.set(assignedSeat, msg.payload)
 // ...
}
```

**修复后**(先扫 peers找同 uuid,找到则复用 seat;否则正常分配):

```js
// 网络层内部模块级变量(默认走原生 API,测试时替换)
let _setIntervalFn = setInterval
let _clearIntervalFn = clearInterval
let _setTimeoutFn = setTimeout
let _clearTimeoutFn = clearTimeout

// UUID持久化 key(sessionStorage 而非 localStorage,刷新复用,关浏览器失效)
const UUID_KEY = 'guandan_session_uuid'

function ensureUuid() {
 try {
 let u = sessionStorage.getItem(UUID_KEY)
 if (!u) {
 u = (typeof crypto !== 'undefined' && crypto.randomUUID)
 ? crypto.randomUUID()
 : 'u-' + Math.random().toString(36).slice(2) + Date.now().toString(36)
 sessionStorage.setItem(UUID_KEY, u)
 }
 return u
 } catch (e) {
 // sessionStorage不可用(测试环境/隐私模式)
 return 'u-' + Math.random().toString(36).slice(2) + Date.now().toString(36)
 }
}

function startAsHost(self) {
 selfInfo = { ...self, uuid: ensureUuid() } // ★ 自动注入 uuid
 isHostFlag = true
 selfSeat =0
 peers.set(0, { ...selfInfo }) // ★ 自己入 peers 也带 uuid
 // ...
 startHeartbeatChecker() // ★启动 host 心跳检查
 // ...
}

function joinRoom(hostRoomId, self) {
 selfInfo = { ...self, uuid: ensureUuid() } // ★ 自动注入 uuid
 isHostFlag = false
 selfSeat = -1
 roomId = hostRoomId
 // ...
 startHeartbeat() // ★启动 joiner 心跳发送
 // ...
}

// host收 JOIN 处理(改造)
if (msg.type === 'JOIN') {
 const newUuid = msg.payload?.uuid
 // ★ Bug1修复:扫 peers找同 uuid →复用 seat
 let assignedSeat = -1
 if (newUuid) {
 for (const [seat, p] of peers.entries()) {
 if (seat ===0) continue //房主位不让
 if (p?.uuid === newUuid) { assignedSeat = seat; break }
 }
 }
 if (assignedSeat !== -1) {
 //复用旧 seat,更新 nickname/avatar/ready
 const updated = { ...peers.get(assignedSeat), ...msg.payload, uuid: newUuid }
 peers.set(assignedSeat, updated)
 emit('peer:update', { seat: assignedSeat, info: updated })
 sendMessage({
 type: 'SYNC',
 payload: { peers: Array.from(peers.entries()) },
 to: msg.from,
 })
 return // ★走完,不分配新 seat
 }
 // 否则正常找下一个空位
 const used = new Set(Array.from(peers.keys()))
 for (let i =1; i <4; i++) {
 if (!used.has(i)) { assignedSeat = i; break }
 }
 if (assignedSeat === -1) {
 sendMessage({ type: 'ROOM_FULL', payload: { reason: '房间已满' }, to: msg.from })
 return
 }
 peers.set(assignedSeat, msg.payload)
 emit('connect', { seat: assignedSeat, info: msg.payload })
 sendMessage({
 type: 'SYNC',
 payload: { peers: Array.from(peers.entries()) },
 to: msg.from,
 })
}
```

**joiner端 SYNC 处理**(找自己 seat 的逻辑从 `nickname`改为 `uuid`):
```js
if (msg.type === 'SYNC' && msg.payload.peers) {
 for (const [seat, info] of msg.payload.peers) peers.set(seat, info)
 let assignedSeat = -1
 // ★ Bug1修复:用 uuid找自己(nickname 可能重复,uuid不会)
 for (let i =1; i <4; i++) {
 const p = peers.get(i)
 if (p && p.uuid === selfInfo.uuid) { assignedSeat = i; break }
 }
 if (assignedSeat === -1) {
 // ★ Bug3 处理:并发撞座导致 SYNC 里没我,300ms 后重发 JOIN(去抖)
 scheduleJoinRetry()
 return
 }
 selfSeat = assignedSeat
 peers.set(selfSeat, selfInfo)
 emit('connect', { seat: selfSeat })
}
```

**为什么 UUID走 sessionStorage 而不是 localStorage**:
-刷新 tab → UUID还在 → host识别同一个人 →复用旧 seat(✓ Bug1修复)
- 关浏览器再开 → UUID失效 → 被识别为新玩家(✓符合产品决策:完全退出才视为新玩家)
-走 localStorage的话,跨周打开也会被识别为同一个人,旧座位永远占着(❌)

###3.2 Bug2 — 心跳 +断线检测(P1 高)

**根因**:
- joiner 关 tab 后,host 的 `peers` map永远不清,房间变「假满」,新玩家进不来
- 当前没有心跳机制,host不知道 joiner 是否还活着

**修复后**(心跳发送 +接收 + 超时释放):

```js
// 模块级变量
const HEARTBEAT_INTERVAL_MS =3000 // joiner 发心跳间隔
const HEARTBEAT_CHECK_INTERVAL_MS =5000 // host 检查间隔
const HEARTBEAT_TIMEOUT_MS =10000 // host 认为 joiner断线的阈值
let heartbeatSendTimer = null // joiner端
let heartbeatCheckTimer = null // host端
const lastHeartbeat = new Map() // host端:seat -> ts

function startHeartbeat() {
 if (heartbeatSendTimer) return
 heartbeatSendTimer = _setIntervalFn(() => {
 if (!channel) return
 sendMessage({ type: 'HEARTBEAT', payload: { ts: Date.now() } })
 }, HEARTBEAT_INTERVAL_MS)
}

function stopHeartbeat() {
 if (heartbeatSendTimer) {
 _clearIntervalFn(heartbeatSendTimer)
 heartbeatSendTimer = null
 }
}

function startHeartbeatChecker() {
 if (heartbeatCheckTimer) return
 heartbeatCheckTimer = _setIntervalFn(() => {
 const now = Date.now()
 for (const [seat, ts] of lastHeartbeat.entries()) {
 if (now - ts > HEARTBEAT_TIMEOUT_MS) {
 // ★ 超时释放
 lastHeartbeat.delete(seat)
 const info = peers.get(seat)
 peers.delete(seat)
 emit('peer:leave', { seat, info })
 // ★广播通知其它 joiner(可选,SYNC也能起到同样效果)
 sendMessage({ type: 'PEER_LEAVE', payload: { seat } })
 }
 }
 }, HEARTBEAT_CHECK_INTERVAL_MS)
}

function stopHeartbeatChecker() {
 if (heartbeatCheckTimer) {
 _clearIntervalFn(heartbeatCheckTimer)
 heartbeatCheckTimer = null
 }
 lastHeartbeat.clear()
}

// host收 HEARTBEAT(在 channel.onmessage 内 JOIN / NICK_UPDATE / READY 分支外)
} else if (msg.type === 'HEARTBEAT') {
 if (peers.has(msg.from)) {
 lastHeartbeat.set(msg.from, Date.now())
 }
}

// joiner收 PEER_LEAVE
} else if (msg.type === 'PEER_LEAVE') {
 const seat = msg.payload?.seat
 if (seat != null && peers.has(seat)) {
 peers.delete(seat)
 emit('peer:leave', { seat })
 }
}

// close()改造
function close() {
 stopHeartbeat() // ★ joiner 清心跳发送
 stopHeartbeatChecker() // ★ host 清心跳检查
 if (channel) { try { channel.close() } catch (e) {} channel = null }
}
```

**为什么不双向心跳**(host 也发,joiner 也检查):
- v1.0 单局跑完 host 不主动离开,joiner断线检测意义不大
- v2.0 真机版再做双向(v1.0 spec §六 已留口)

**为什么不立即检测**(joiner 关 tab 后立刻释放 seat):
-浏览器关 tab 不发「退出」事件,host 无法立刻知道
-10s 超时是合理容错(网络抖动 + 用户切换 tab 不至于误判)

###3.3 Bug3 — 并发 joiner撞座(P1 中)

**根因**:
-2 个 joiner 同时加载,host收2 个 JOIN,各自分配不同 seat(本应正常)
- 但 joiner端 SYNC 处理反查自己 seat 时,如果 SYNC 在另一个 joiner之前到达,**可能找不到自己**(因为 host还没来得及把第二个 joiner 加进 peers)→第二个 joiner 的 SYNC 没带「第二个 joiner 的 uuid」(还没收到 JOIN)
-实际撞座更可能的场景:joiner A 先收到 host 的 SYNC(只有 host 自己),反查自己 seat失败;joiner B 也收到同样的 SYNC,反查自己 seat 也失败。两者各自走「重发 JOIN」流程,但第二次 JOIN间隔极短,可能 host还没处理第一次就收到第二次

**简化方案(推荐,本 spec 采用)**:

joiner收到 SYNC找不到自己时,**300ms 后重发 JOIN**(去抖,确保 host收到2 次 JOIN 后再分配)。host收 JOIN 时:
-第一次:正常分配 seat
-第二次(同 uuid短时间内):已存在的 seat,更新 info(走 Bug1 的复用路径)

**修复后 joiner端**:

```js
let joinRetryTimer = null // 去抖

function scheduleJoinRetry() {
 if (joinRetryTimer) return // ★ 去抖:已有定时器,不再排
 joinRetryTimer = _setTimeoutFn(() => {
 joinRetryTimer = null
 if (!channel) return
 // ★ 重发 JOIN,host收到后会重新分配 seat(走 Bug1复用路径)
 sendMessage({ type: 'JOIN', payload: selfInfo })
 },300)
}

function cancelJoinRetry() {
 if (joinRetryTimer) {
 _clearTimeoutFn(joinRetryTimer)
 joinRetryTimer = null
 }
}

// SYNC 处理(改造)
if (msg.type === 'SYNC' && msg.payload.peers) {
 for (const [seat, info] of msg.payload.peers) peers.set(seat, info)
 let assignedSeat = -1
 for (let i =1; i <4; i++) {
 const p = peers.get(i)
 if (p && p.uuid === selfInfo.uuid) { assignedSeat = i; break }
 }
 if (assignedSeat === -1) {
 scheduleJoinRetry() // ★ Bug3修复:并发撞座时 retry
 return
 }
 cancelJoinRetry() // ★成功分配,清掉 retry定时器
 selfSeat = assignedSeat
 peers.set(selfSeat, selfInfo)
 emit('connect', { seat: selfSeat })
}

// close()改造(已有,补一行)
function close() {
 stopHeartbeat()
 stopHeartbeatChecker()
 cancelJoinRetry() // ★ 新增
 if (channel) { try { channel.close() } catch (e) {} channel = null }
}
```

**为什么不优化为「host主动广播 SYNC 给所有 joiner」**:
- 当前 P0修完后 host 只给新 joiner 发 SYNC(to=msg.from),其它 joiner 不收
- 如果改成 broadcast,会触发 Bug2 已修的过滤问题(sendTo to字段被忽略),且每个 joiner都要重处理 SYNC
- 当前 joiner主动重发 JOIN + host走 Bug1复用路径,改动最小

**为什么不阻塞 joiner 等 seat分配**:
- 当前是事件驱动,joiner收到 SYNC 就 `emit('connect')`,如果走 Promise化会破坏现有 API
-300ms 重试是异步、非阻塞的,符合事件驱动原则

## 四、RoomView.vue 可选改的1 行(纯 UI体验,非硬依赖)

**为什么可以不改**:
- `network.js` 在 emit('peer:leave') 后,host内部会广播 PEER_LEAVE →其它 joiner收到后 emit('peer:leave')
- `RoomView.vue`已经有 `net.on('message:SYNC', ...)` 清 peers,但 **PEER_LEAVE 单个释放不会触发 SYNC**
- 不改 → UI 上某个 seat 显示「等待加入」需要等 SYNC再次广播才更新(可能延迟)
-改了 → UI 即时释放 seat

**修复后**(L143 `initNetwork`末尾加):
```js
net.on('peer:leave', ({ seat }) => {
 // ★ P1 可选:PEER_LEAVE事件 UI同步释放 seat
 if (peers.has(seat)) peers.delete(seat)
})
```

**为什么标「可选」**:
- 不改不影响 P1 bug修复目标(UUID 去重 / 心跳 / 并发撞座都已验证)
-改是为了 UI流畅,verifier跑 Playwright 时会观察到这个差异
-严格按「别动网络层以外的文件」,不强制改

**fake timer 实现**(跨实例):
每个 module instance内部用 `_setIntervalFn`,因为 module-level变量每个 instance独立(ESM缓存被 query string破),所以可以**在 instance自己的闭包里替换**。最简单做法:每个测试 instance替换自己的 `_setIntervalFn` 为可控函数,直接调 callback:

```js
async function makeFakeInstance(tag) {
 const mod = await loadFresh(tag)
 const callbacks = []  //收集所有 setInterval 注册的 callback
 mod._setIntervalFn = (fn, ms) => {
 callbacks.push({ fn, ms })
 return callbacks.length //假 id
 }
 mod._clearIntervalFn = (id) => {
 if (id >=1 && id <= callbacks.length) callbacks[id -1].cancelled = true
 }
 return { mod, callbacks }
}

// 测试时:

// Host内部调 startHeartbeatChecker,注册了 setInterval

//模拟「时间走过」:直接调 callback

//断言 lastHeartbeat / peers变化
```

**为什么不用 `vi.useFakeTimers` / `jest.useFakeTimers`**:
- 项目零依赖,不引入 vitest / jest
- 用 callback列表 +手动调是最低成本方案

## 八、风险点

1. **fake timer 实现风险**:
 - 用「回调列表 +手动调 callback」模拟时间走过,**漏调 callback = 测试假绿**
 -缓解:`network.test.js` §17 必须显式断言「调 callback 后状态变化」(不是「不调 →状态不变」)
 -缓解:verifier跑测试时单独检查 `npm test` 输出,确认 timer callback确实被调

2. **sessionStorage 在测试环境不可用**:
 - Node 单测环境没有 `sessionStorage`(浏览器 API)
 - 当前方案:`network.js`内部 try/catch,fallback 到 `Math.random()` 生成 UUID
 - 测试时 mock `globalThis.sessionStorage = { getItem: () => 'fixed-uuid', setItem: () => {} }` 让 UUID固定
 -风险:忘了 mock → UUID每次随机,「同 uuid 重连」测试假绿
 -缓解:测试用例显式 mock + assert `selfInfo.uuid === 'fixed-uuid'`

3. **并发撞座 retry死循环**:
 -极端场景:host满员(4 个)后,新 joiner SYNC找不到自己 → retry →收到 ROOM_FULL → 没处理 retry
 - 当前:retry 只发 JOIN,joiner端收到 ROOM_FULL走现有 error 处理,不会无限 retry
 -验证:`network.test.js` §20 + §21 必须断言「retry1 次后 stop」(去抖生效)

4. **心跳 setInterval泄漏(close 未调)**:
 - 测试 §19 必须断言 `_clearIntervalFn` 被调,否则即使 close()漏清 timer 也通过不了测试
 -缓解:RoomView.vue L195 `onUnmounted(() => net.close())` 已调 close,只要 close()内部清就行

5. **HEARTBEAT消息被业务订阅**:
 - `net.on('message:HEARTBEAT')`会被业务方误订阅
 -缓解:he artbeat 是基础设施消息,业务方应该监听 `peer:leave` 而不是 `message:HEARTBEAT`(NETWORK.md §3.1 已定义 peer:leave事件)
 -文档:NETWORK.md §三 加一行「HEARTBEAT 是基础设施消息,业务层勿订阅」

6. **P0死锁回归风险**:
 - Bug1修复后 host JOIN 处理变长,可能破坏 P0 的自洽性
 -缓解:`network.test.js` §11-13验证 JOIN payload 含 uuid / host收 JOIN走 Bug1路径
 -缓解:P0 的38 个用例**全保留**,任何修改不能破坏

7. **GameView.vue 不调 net.close()**(已知,不在本轮):
 - 见 §〇副作用 D
 - 影响:Vue Router缓存 RoomView 时 channel 不关
 - 本轮不修,留给后续 plan(可能是 RoomView/GameView路由生命周期专项)

8. **uuid字段加入 peers map 后,旧测试断言可能 break**:
 - `network.test.js` §1断言 `peers[0].nickname = '房主'` ——修后仍成立(只是 entry 多一个 uuid字段)
 - §8断言 `peers.set(1, { nickname: 'P1', avatar: '1' })` ——修后**手造 entry 不带 uuid**,但 Bug1修复路径找 uuid,**手造 entry永远走「新分配」路径**
 -缓解:§8改成手造 entry 带 uuid,或者断言改成「host收4 号 joiner JOIN 后,host emit ROOM_FULL」绕过 uuid匹配

9. **dev mode HMR 影响**(沿用 P0风险):
 - Vite HMR 重载模块时,旧的 channel.onmessage handler 被 GC
 -缓解:verifier跑测试时建议 `npm run build && npm run preview` 或 HMR 时手动刷新所有 tab
 - 影响范围:仅 dev,prod 不受影响
