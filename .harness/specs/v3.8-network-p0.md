## 三、3 个 P0 的具体改法

### 3.1 Bug 1 — 死锁(P0 致命)

**根因**:
- `joinRoom` 创建 channel 后**啥也不发**,死等 host 的 SYNC
- `startAsHost` 创建 channel 后**啥也不广播**,死等 joiner 的 JOIN
- 双方互相等 → 永远无消息

**修复前**(network.js L78-114,核心片段):

```js
function joinRoom(hostRoomId, self) {
  // ...
  channel = new BroadcastChannel('guandan-p2p-' + hostRoomId)
  channel.onmessage = (event) => {
    const msg = event.data
    if (!msg || msg.from === selfSeat) return
    // ... 等 SYNC ...
    if (msg.type === 'SYNC' && msg.payload.peers) {
      // 找空位,分配 seat
      // ... 然后才发 JOIN
      sendMessage({ type: 'JOIN', payload: self })
    }
  }
  return { ok: true }
}
```

**修复后**:

```js
function joinRoom(hostRoomId, self) {
  // ...
  channel = new BroadcastChannel('guandan-p2p-' + hostRoomId)
  channel.onmessage = (event) => {
    const msg = event.data
    if (!msg || msg.from === selfSeat) return
    if (msg.to != null && msg.to !== selfSeat) return   // ★ Bug 2 修复(对称)
    // ... 
    if (msg.type === 'SYNC' && msg.payload.peers) {
      // host 已经把 joiner 占的 seat 存进 peers,反查自己的 nickname 找 seat
      for (let i = 0; i < 4; i++) {
        if (peers.get(i)?.nickname === self.nickname) { selfSeat = i; break }
      }
      if (selfSeat === -1) { emit('error', '座位分配失败'); return }
      peers.set(selfSeat, self)  // 覆盖 host 占位,刷新成自己最新 info
      emit('connect', { seat: selfSeat })
      // ★ 不再发 JOIN,host 已经知道我们
    }
  }
  // ★ Bug 1 修复:创建 channel 后立即发 JOIN,host 收到后回 SYNC
  sendMessage({ type: 'JOIN', payload: self })
  return { ok: true }
}
```

**host 端对应的 seat 分配改动**(L51-74):

```js
// 修复前:
if (msg.type === 'JOIN') {
  peers.set(msg.from, msg.payload)                      // ← msg.from = -1,存错 key
  emit('connect', { seat: msg.from, info: msg.payload }) // ← seat = -1,UI 也错
  sendMessage({ type: 'SYNC', payload: { peers: Array.from(peers.entries()) }, to: msg.from })
}

// 修复后:
if (msg.type === 'JOIN') {
  // host 主动分配 seat(joiner 报 from=-1,自己不算,host 找下一个空位)
  const used = new Set(Array.from(peers.keys()))
  let assignedSeat = -1
  for (let i = 1; i < 4; i++) {
    if (!used.has(i)) { assignedSeat = i; break }
  }
  if (assignedSeat === -1) {
    sendMessage({ type: 'ROOM_FULL', payload: { reason: '房间已满' }, to: msg.from })
    return
  }
  peers.set(assignedSeat, msg.payload)
  emit('connect', { seat: assignedSeat, info: msg.payload })
  sendMessage({
    type: 'SYNC',
    payload: { peers: Array.from(peers.entries()) },
    to: msg.from,    // msg.from = -1,joiner 收到时 to===selfSeat(-1) 仍能继续
  })
}
```

**自洽性检查**(PM 已验):
- joiner 发的 JOIN `from = -1`
- joiner 端 onmessage 看到 `from === selfSeat` 即 `-1 === -1` → **return**(joiner 不处理自己发的)✓
- host 端 onmessage 看到 `from === selfSeat` 即 `-1 === 0` → **不 return**(继续)✓
- host 回 SYNC `to: -1`
- joiner 端 onmessage 看到 `to != null && to !== selfSeat` 即 `-1 !== -1` → **不 return**(继续)✓
- 修复后 joiner 收到 SYNC 时 selfSeat 仍为 -1,反查 nickname 找到 host 分配的 seat 后才更新

### 3.2 Bug 2 — `sendTo` 不过滤 `to`(P0 高)

**根因**:`sendTo(seat, payload)`(L118)加了 `to: seat`,但 host/joiner 的 `onmessage` 顶部(L51-55 / L88-92)**不看 `msg.to`**,所有消息都当自己的处理。

**修复前**(host 端 L51-53):
```js
channel.onmessage = (event) => {
  const msg = event.data
  if (!msg || msg.from === selfSeat) return
  emit('message', msg)            // ← 不看 to,广播消息误触发
  emit('message:' + msg.type, ...) // ← 定向消息误触发
  // ...
}
```

**修复后**(host + joiner **对称**修改):
```js
channel.onmessage = (event) => {
  const msg = event.data
  if (!msg || msg.from === selfSeat) return
  if (msg.to != null && msg.to !== selfSeat) return   // ★ Bug 2 修复
  emit('message', msg)
  emit('message:' + msg.type, msg.payload, msg.from, msg)
  // ...
}
```

**为什么用 `!= null`**:同时挡掉 `null` 和 `undefined`(`to` 字段从未设置时 `msg.to` 是 `undefined`)。
**为什么对称**:joiner 端如果不加,Bug 1 修复后 joiner 收到的 SYNC/ROOM_FULL `to: -1` 会被忽略,死锁复活。

### 3.3 Bug 3 — `scanLanRooms` 伪造数据(P0 中)

**根因**:L126-130 始终返回 `[{ ip: '127.0.0.1', name: '本机(广播通道)' }]`,误导用户以为真的有 LAN 扫描。

**修复前**:
```js
async function scanLanRooms() {
  return [
    { ip: '127.0.0.1', name: '本机(广播通道)' },
  ]
}
```

**修复后**:
```js
/**
 * 扫描局域网房间
 * @returns {Promise<Array<{ip: string, name: string}>>}
 *
 * v1.0 浏览器版不支持真实 LAN 扫描(BroadcastChannel 只在同 origin 同浏览器互通),
 * 始终返回空数组。真实 LAN 扫描需 v2.0 接入 TCP 平台(见 docs/NETWORK.md §6)。
 *
 * JoinView 收到 [] 时,UI 应显示「未发现房间,请让房主把房间号发你」并允许手动输入房间号。
 */
async function scanLanRooms() {
  return []
}
```

**配套 UI 校验**(JoinView.vue):不用改代码,确认 UI 在 `rooms.length === 0` 时显示手动输入入口即可(原 UI 应该已有,verifier 截图验收)。

### 5.2 跨实例测试(`network-multitab.test.js` 新增)

**核心技巧**:用 dynamic import + query string 破 ESM module 缓存,让每个 import 都是独立的 module evaluation(独立的 `selfSeat / channel / peers`)。同 process 内相同 name 的 `BroadcastChannel` 互通(Node 18+ 全局 `BroadcastChannel`)。

```js

async function loadFresh(tag) {
  const url = `./network.js?tag=${tag}&t=${Date.now()}_${Math.random()}`
  return await import(url)
}

console.log('\n=== 1. host 分配 seat ===')
const Host = await loadFresh('host')
Host.setRoomId('room-test-1')
Host.startAsHost({ nickname: '房主', avatar: '🀄' })

const Joiner1 = await loadFresh('j1')
Joiner1.joinRoom('room-test-1', { nickname: 'A', avatar: '♠' })

// 等待 BroadcastChannel 异步分发(setTimeout 50ms 通常够)
await new Promise(r => setTimeout(r, 100))

assert('joiner1 拿到 seat 1', Joiner1.getSelfSeat() === 1)
assert('host peers 含 joiner1 seat 1', Host.getPeers().get(1)?.nickname === 'A')
```

**测试块设计**:

| 块 | 测什么 | 涉及 bug |
|---|---|---|
| 1. host 分配 seat | joiner 立即收到 seat,host peers 正确 | bug 1 |
| 2. 3 个 joiner 排队 | 第二个进 seat 2,第三个进 seat 3 | bug 1 |
| 3. 第 4 个 joiner | 进不来,host 收到 ROOM_FULL,joiner 收到 error | bug 1 副作用 |
| 4. **sendTo 过滤 to** | host sendTo(2, X) → 只有 joiner2 触发,joiner1/3 不触发 | bug 2 跨实例 |

## 八、风险点

1. **BroadcastChannel 浏览器兼容性**:
   - Chrome 54+ / Firefox 38+ / Safari 15.4+ 才支持
   - **iOS Safari 15.4 之前的用户会直接报 "BroadcastChannel 不支持"**(见 `network.js` L49 / L86 的 try/catch)
   - 当前 v1.0 已 try/catch,失败回 `{ ok: false, error }`,RoomView 收到后 netStatus = 🔴 —— 行为可控
   - 真机跨设备联机本来就是 v2.0 范畴(见 `NETWORK.md` §6.4),不在 P0 修复范围

2. **dev mode HMR 影响 BroadcastChannel**:
   - Vite HMR 重载模块时,旧的 `channel.onmessage` handler 会被 GC,新的 handler 注册后**收不到 reload 期间其它 tab 发的消息**
   - 缓解:测试时建议 `npm run build && npm run preview` 或 HMR 时手动刷新所有 tab
   - 影响范围:仅 dev,prod 不受影响

3. **跨实例测试稳定性**:
   - dynamic import + query string 破缓存依赖 Node 18+ 的 ESM loader 行为
   - `BroadcastChannel` 在同 process 不同 module instance 间互通(已实测:Node 20.10 OK)
   - 建议 verifier 跑测试时显式确认 `node -v` ≥ 18

4. **RoomView 顺序未改就过 review**:
   - §四 的 1 行调换是 4-tab 验收硬依赖,reviewer 必查
   - 建议 verifier 在 PR diff 里 grep `setRoomId` 必须在 `startAsHost` 之前

5. **seat 分配冲突**:
   - 当前 host 端只在收到 JOIN 时找空位,**没处理"joiner 突然 close"释放 seat** —— v3.8 不修
   - v1.0 单局跑完 4-tab 不会动态进出,影响小;v2.0 真机联机再做 seat 回收

6. **`scanLanRooms` 返回 `[]` 影响 JoinView 现有 UI**:
   - 如果 JoinView 当前假设至少有 1 项(`rooms[0].ip`),UI 会 NPE
   - 缓解:verifier 跑 §5.3 时专门截图 JoinView 空状态,确认无报错
